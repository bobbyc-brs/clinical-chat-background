# Project Notes

## What this project does

Captures a doctor/patient conversation, transcribes it, and uses an LLM to automatically fill a medical intake form. It also generates follow-up questions for anything missing or unclear.

---

## What is hardcoded

The demo conversation in `clinical-server/clinical_server.py` inside the `/api/demo/load` endpoint — 6 pre-written lines of doctor/patient dialogue with fake confidence scores. This is only for demo purposes and has no bearing on the real pipeline.

---

## Where the LLM acts

After the transcript is stored, `llm_extract()` sends two things to the LLM:

**Input:**
- The form field definitions from `clinical-server/form_template.yaml` (field keys, labels, synonyms)
- The full conversation timeline with speaker roles and confidence scores

**Output:**
- Each form field filled with: extracted value, confidence score, evidence (the transcript line used), and status
- A list of follow-up questions for fields that are missing or unclear

---

## How the UI looks

**Before loading demo data** — empty dashboard, all fields missing, no timeline, no follow-ups

**After clicking "Load demo data"** — three panels populate:
- Left: conversation timeline with doctor/patient turns color-coded
- Middle: medical form with fields marked filled / review / missing
- Right: follow-up prompts generated by the LLM for what it couldn't extract

## Known bugs

**Bug 1: Race condition in `/api/demo/load`**
`load_demo()` calls `process_batch()` twice — once for doctor, once for patient. Each call immediately triggers `llm_extract()`, so the LLM is called before both speakers' data is in the timeline. Result: first LLM call only sees doctor lines, second call sees both. The form gets partially filled then overwritten.

**Bug 2: UI doesn't use the demo response**
The dashboard `await`s the POST to `/api/demo/load` but discards the response, then immediately calls `render()` which fetches `/api/sessions/visit-001`. Because the LLM takes a few seconds, `render()` often runs before extraction is complete — requiring the user to manually refresh 1-2 times to see the final result.

**Bug 3: Status label mismatch**
The LLM returns `"complete"` as the field status but the UI expects `"filled"`. Fields may not render with the correct color/badge.

---

## What is missing for full usability

| Missing | Impact |
|---|---|
| Real mic capture — edge client broken (`fspyrpc` package is broken) | No live transcription, stuck on hardcoded demo data |
| No session persistence | Restart the server = all data lost |
| No session reset endpoint | Calling demo twice accumulates duplicate timeline entries |
| No authentication | Anyone on the network can access patient data |
| Single hardcoded session `visit-001` | Cannot run multiple concurrent patient visits |

---

## Current setup

- LLM provider: OpenRouter (env var `OPENROUTER_API_KEY`)
- Default model: `cohere/north-mini-code:free` (env var `OPENROUTER_MODEL`)
- Server runs on port `8080` (Flask)
- RPC listener on port `9090` (non-functional, `fspyrpc` broken)
- State is in-memory only — no database

## Fix applied during hackathon exploration

The original code used `response_format: json_schema` which the Cohere model ignores. Fixed in `llm_extract()` by embedding the schema in the system prompt and stripping markdown code fences before parsing the response.
